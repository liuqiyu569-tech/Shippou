# 开发指南

本指南用于统一开发流程，确保代码质量和工程规范。

环境初始化与联调准备请先阅读：`docs/环境配置说明.md`。

## 1. Git 规范

### 1.1 分支模型
- main：可发布、可验收分支
- develop：日常集成分支
- feature/*：功能开发分支
- hotfix/*：线上紧急修复分支

创建方式：
- 新功能：从 develop 拉 feature/xxx
- 热修复：从 main 拉 hotfix/xxx

### 1.2 Commit Message 规范
推荐 Conventional Commits：
- feat: 新功能
- fix: 缺陷修复
- refactor: 重构（不改行为）
- docs: 文档变更
- test: 测试相关
- chore: 构建/工具/配置

要求：
- 标题不超过 72 字符
- 描述清晰说明改动意图
- 禁止使用"update""modify"这类无信息标题

### 1.3 审查清单
提交前自查：
- 本次改动可独立运行
- 无调试垃圾代码和无用日志
- 接口返回结构未破坏约定
- 错误处理和提示信息完整
- 关键逻辑有注释或文档说明

## 2. 编码规范

### 2.1 命名规范
- 类名：大驼峰，如 TaskService
- 方法/变量：小驼峰，如 getTaskById
- 常量：全大写下划线，如 MAX_PAGE_SIZE
- 枚举：使用语义化值，禁止魔法数字

### 2.2 结构与复杂度
- 单一职责：一个类只做一类事
- 一个方法只完成一个明确动作
- 使用卫语句减少多层 if 嵌套
- 复用公共逻辑，避免复制粘贴

### 2.3 注释与文档
- 注释用于解释设计意图，不翻译代码
- 对核心类的 public 方法添加 JavaDoc/TS Doc
- 遇到技术债使用 TODO 并标明原因与计划

### 2.4 排版与风格
- 单行不超过 120 字符
- 统一缩进（后端 4 空格，前端遵循 prettier）
- 类成员、方法之间保留空行增强可读性

## 3. 安全与数据规范
- 密码必须使用 BCrypt 哈希存储
- 任何任务操作都必须校验 user_id = 当前登录用户
- 不在日志中打印密码、token 等敏感信息
- 后端统一返回错误结构，避免泄露内部堆栈

## 4. API 与前后端联调规范
- 所有接口前缀统一（例如 /api）
- 请求/响应使用统一 DTO，不直接暴露实体
- 前端通过统一 http 客户端注入 token
- 401 统一跳转登录页，403 统一提示无权限
