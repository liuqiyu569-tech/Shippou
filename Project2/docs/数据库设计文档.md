# 协作式任务管理系统（TaskFlow）数据库设计文档

## 1. 设计目标

本设计在已有数据库结构基础上进行增量扩展，新增 `task_operation_logs` 表，用于支持任务操作留痕功能。

系统不改变已有核心表结构，继续沿用以下表：

1. `users`
2. `teams`
3. `team_members`
4. `tasks`
5. `task_assignees`
6. `task_dependencies`

新增一张日志表：

7. `task_operation_logs`（新增）

该表用于记录任务相关关键操作，包括创建任务、修改任务信息、修改任务状态、修改任务负责人、新增或删除任务依赖、删除任务，以及成员离开或被移除后导致的任务负责人变化。

---

## 2. 实体关系概览

`task_operation_logs` 是任务操作历史表，一条日志对应一次成功的任务相关操作。

- 一个任务可以对应多条操作日志。
- 一个用户可以产生多条操作日志。
- 一个团队可以对应多条团队任务日志。
- 个人任务日志的 `team_id`、`team_name` 为空。
- 团队任务日志的 `team_id`、`team_name` 记录该任务所属团队。

日志表保存部分冗余快照字段，例如 `operator_username`、`task_title`、`team_name`。这样做是为了保证任务被删除、团队名称变更或成员关系变化后，系统仍能合理展示历史日志。

---

## 3. 数据表设计

### 3.1 task_operation_logs（任务操作日志表）

该表用于记录任务相关操作历史。日志只由后端在业务操作成功后自动写入，普通用户不允许修改或删除。

| 字段名 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | BIGINT | PK, AUTO_INCREMENT | 日志主键 |
| operator_id | BIGINT | NOT NULL | 操作人 ID，来自当前登录用户 |
| operator_username | VARCHAR(50) | NOT NULL | 操作人用户名快照 |
| operation_type | VARCHAR(20) | NOT NULL | 操作类型：CREATE / UPDATE / DELETE |
| object_type | VARCHAR(30) | NOT NULL | 操作对象类型：TASK_INFO / TASK_DEPENDENCY / TASK_ASSIGNEE |
| scope_type | VARCHAR(20) | NOT NULL | 操作范围：PERSONAL / TEAM |
| task_id | BIGINT | NOT NULL | 被操作任务 ID |
| task_title | VARCHAR(100) | NOT NULL | 被操作任务标题快照 |
| team_id | BIGINT | NULL | 所属团队 ID，个人任务为空 |
| team_name | VARCHAR(100) | NULL | 所属团队名称快照，个人任务为空 |
| summary | VARCHAR(500) | NOT NULL | 面向用户展示的简要变更说明 |
| operation_time | DATETIME | NOT NULL | 操作发生时间 |

关键约束：
- `operation_type` 只允许 `CREATE`、`UPDATE`、`DELETE`，在 Service 层或枚举中校验。
- `object_type` 只允许 `TASK_INFO`、`TASK_DEPENDENCY`、`TASK_ASSIGNEE`，在 Service 层或枚举中校验。
- `scope_type` 只允许 `PERSONAL`、`TEAM`，在 Service 层或枚举中校验。
- `scope_type = PERSONAL` 时，`team_id` 和 `team_name` 应为空。
- `scope_type = TEAM` 时，`team_id` 和 `team_name` 应记录团队快照。

索引建议：
- `INDEX idx_task_operation_logs_task_time (task_id, operation_time)`：按任务查询日志并按时间排序。
- `INDEX idx_task_operation_logs_operator_time (operator_id, operation_time)`：查询某个用户产生的日志。
- `INDEX idx_task_operation_logs_team_time (team_id, operation_time)`：查询某个团队下的任务日志。
- `INDEX idx_task_operation_logs_scope_time (scope_type, operation_time)`：按个人/团队范围查询日志。
- `INDEX idx_task_operation_logs_type_time (operation_type, object_type, operation_time)`：按操作类型、对象类型筛选日志。

## 4. 日志记录范围

系统需要记录以下任务相关操作：

| 操作场景 | operation_type | object_type | scope_type |
|---|---|---|---|
| 创建个人任务 | CREATE | TASK_INFO | PERSONAL |
| 修改个人任务基本信息或状态 | UPDATE | TASK_INFO | PERSONAL |
| 删除个人任务 | DELETE | TASK_INFO | PERSONAL |
| 新增个人任务依赖 | CREATE | TASK_DEPENDENCY | PERSONAL |
| 删除个人任务依赖 | DELETE | TASK_DEPENDENCY | PERSONAL |
| 创建团队任务 | CREATE | TASK_INFO | TEAM |
| 修改团队任务基本信息或状态 | UPDATE | TASK_INFO | TEAM |
| 删除团队任务 | DELETE | TASK_INFO | TEAM |
| 修改团队任务负责人 | UPDATE | TASK_ASSIGNEE | TEAM |
| 取消团队任务负责人 | DELETE | TASK_ASSIGNEE | TEAM |
| 新增团队任务依赖 | CREATE | TASK_DEPENDENCY | TEAM |
| 删除团队任务依赖 | DELETE | TASK_DEPENDENCY | TEAM |
| 成员离开导致任务负责人变化 | DELETE | TASK_ASSIGNEE | TEAM |
| 成员被移除导致任务负责人变化 | DELETE | TASK_ASSIGNEE | TEAM |

团队解散后任务状态或访问方式变化不作为必做日志记录范围。

---

## 5. 关键业务规则（在 Service 层强校验）

1. 日志只能由后端在业务操作成功后自动生成，前端不能直接提交操作人、日志内容或日志时间。
2. 操作失败时，不写入成功日志。例如权限不足、参数非法、依赖关系不合法、任务不存在等情况均不应生成成功操作日志。
3. 日志中不得记录用户密码、Token、会话密钥等敏感信息。
4. 创建任务日志建议记录任务标题、初始状态、优先级和截止时间，不建议完整记录任务描述。
5. 修改任务日志只记录发生变化的字段，例如状态、优先级、截止时间等。
6. 删除任务前应保存必要快照，例如任务标题、所属团队名称，使任务删除后仍可查看历史日志。
7. 个人任务日志只能由任务所属用户查看。
8. 团队任务日志只能由当前仍属于该团队、且有权查看该任务的用户查看。
9. 已离开或被移除团队的用户，不得继续查看该团队任务日志。
10. 普通用户、团队 Owner 和 Admin 都不允许通过业务接口修改或删除日志。

---

## 6. 删除策略

| 删除操作 | 处理策略 |
|---|---|
| 删除个人任务 | 删除任务本身，但保留该任务相关日志；日志中通过 `task_id`、`task_title` 继续展示历史信息 |
| 删除团队任务 | 删除任务本身，但保留该任务相关日志；日志中通过 `task_id`、`task_title`、`team_id`、`team_name` 继续展示历史信息 |
| 成员离开或被移除 | 清理其任务分配关系，同时为受影响任务写入负责人变化日志 |
| 解散团队 | 按原有删除策略清理团队相关数据；是否记录团队解散导致的任务访问变化不作为必做范围 |
| 删除日志 | 业务系统不提供普通用户删除日志的接口；如后续需要归档或清理，应由系统管理员通过受控后台或定时任务完成 |

---

## 7. 建表示例

```sql
CREATE TABLE task_operation_logs (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    operator_id BIGINT NOT NULL,
    operator_username VARCHAR(50) NOT NULL,
    operation_type VARCHAR(20) NOT NULL,
    object_type VARCHAR(30) NOT NULL,
    scope_type VARCHAR(20) NOT NULL,
    task_id BIGINT NOT NULL,
    task_title VARCHAR(100) NOT NULL,
    team_id BIGINT NULL,
    team_name VARCHAR(100) NULL,
    summary VARCHAR(500) NOT NULL,
    operation_time DATETIME NOT NULL,
    INDEX idx_task_operation_logs_task_time (task_id, operation_time),
    INDEX idx_task_operation_logs_operator_time (operator_id, operation_time),
    INDEX idx_task_operation_logs_team_time (team_id, operation_time),
    INDEX idx_task_operation_logs_scope_time (scope_type, operation_time),
    INDEX idx_task_operation_logs_type_time (operation_type, object_type, operation_time)
);
```

说明：日志表中的 `task_id`、`team_id`、`operator_id` 主要用于查询和追溯。由于任务或团队可能被删除，是否设置外键需要谨慎。为了保证删除任务后日志仍可保留，本设计建议不对 `task_id` 设置强外键约束，或使用不会级联删除日志的外键策略。
